# Telegram Voice MVP

Enable a pragmatic Telegram voice loop: inbound Telegram voice messages get transcribed by OpenClaw via Groq STT, and outbound spoken replies are generated by a standalone Speechify script and sent as Telegram voice notes.

## Quick workflow

1. Read `references/implementation-notes.md` for the exact config shape, validation steps, and known failure modes.
2. For inbound STT, verify OpenClaw `tools.media.audio.scope` includes `telegram`, use Groq `whisper-large-v3-turbo`, and set `prompt` to `Please transcribe verbatim.` to avoid the `Transcribe the audio.` echo bug.
3. For outbound TTS, use `scripts/speechify_tts_probe.py` with a temporary `SPEECHIFY_API_KEY` environment variable. Do not write Speechify secrets into `openclaw.json`.
4. If needed, transcode to Telegram-friendly OGG/Opus and send with `message(action=send, asVoice=true)`.
5. When troubleshooting, follow the checklist in `references/implementation-notes.md` instead of guessing. Most failures were caused by bad prompt/provider combinations, near-silent source audio, or empty TTS output files.

## Guardrails

- Keep `openclaw.json` edits limited to the inbound STT path. Do not touch cron jobs or unrelated tasks.
- Prefer temporary environment variables for provider keys that do not need to live in main config.
- Verify generated audio files are non-empty before sending to Telegram.
- If a new provider is tested inside the container, persist the install steps in `/home/node/openclaw-host/docker-compose.override.yml`.

## Use the bundled script

Run `scripts/speechify_tts_probe.py` for isolated Speechify output checks.

Typical safe flow:

```bash
export SPEECHIFY_API_KEY='...'
python3 /home/node/.openclaw/workspace/skills/telegram-voice-mvp/scripts/speechify_tts_probe.py \
  --text 'Hello Mark, this is a Telegram voice test.' \
  --voice-id george \
  --audio-format mp3 \
  --telegram-voice
```

Then send the resulting `.telegram.ogg` with Telegram `asVoice: true`.
